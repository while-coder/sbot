Agenda system — full decision rules. Call agenda_wiki when uncertain about any of the topics below.

# 1. MENTAL MODEL

An agenda has TWO layers:
- ITEM — the thing being tracked (id, content, status, dueAt). Lifecycle: pending → done / cancelled.
- TRIGGERS — 0..N schedules attached to an item; each has its own kind / expr / action / message / fireCount. Disabling a trigger does NOT cancel the item.

Status of the ITEM is independent of triggers' enabled state. Cancelling the item disables all triggers; disabling triggers does not cancel the item.

# 2. COMPLETION

How an item becomes done depends ONLY on its triggers — there is no completion-mode flag:
- no triggers (plain todo) → stays pending until the user explicitly completes it (agenda_complete).
- has a one-off / finite trigger (one absolute fire, or a count-bounded routine) → auto-closes (status→done) once the schedule is fully exhausted; the user does NOT call complete.
- has an infinite recurring trigger → never auto-closes; ends only via agenda_cancel.

agenda_complete always closes the WHOLE item (status→done, all triggers disabled). For a recurring routine the user keeps doing (drinking water, daily report), do NOT complete on each "did it today" — just acknowledge; the routine keeps firing. Complete only when the whole thing is truly done; stop a routine with agenda_cancel.

# 3. dueAt vs TRIGGER — they are NOT the same thing

dueAt is PURE METADATA:
- Used for: findNearDuplicate dedup key, "overdue" badge, list sort fallback, display.
- Does NOT auto-create any trigger. Does NOT auto-fire anything.
- For a plain todo with a deadline ("finish weekly report by Friday") → set dueAt only.
- For a deadline that should also PING the user → set dueAt AND add a matching absolute trigger.

Changing dueAt does NOT retime existing triggers. If the user moves a deadline AND the existing trigger was the deadline reminder, also call agenda_trigger op=update / op=replace_all with the new schedule. Otherwise the trigger keeps firing at its old time.

# 4. CADENCE CHANGES — cross-tool sync

When the user changes cadence ("every hour" → "every 2 hours", "9am weekdays" → "10am weekdays"):
1. Update the schedule via agenda_trigger op=update / op=replace_all.
2. Inspect whether `content` (item-level) or any other trigger's `message` embeds the OLD cadence text. If so, sync them in the same turn:
   - content → agenda_update with content=...
   - message of other triggers → agenda_trigger op=update on each
3. Skip step 2 only when text is already cadence-agnostic ("submit weekly report" — fine; "submit weekly report every Monday 9am" — needs sync).

Otherwise the displayed name stays stale and contradicts the new schedule.

# 5. SURGICAL vs REPLACE — choosing the trigger op

- Adding ONE trigger alongside existing ones → op=add. ("also remind me a day earlier")
- Editing ONE specific trigger by id → op=update. ("move the 9am reminder to 10am") — read the trigger id from <existing-agenda> XML; the spec must be COMPLETE (existing kind/schedule + new field).
- Disabling ONE trigger → op=remove. ("drop the 18:00 reminder, keep the item")
- The user describes the FULL final list ("from now on only ping at 9 and 18, drop noon") → op=replace_all.
- triggers=[] with op=replace_all clears all active triggers (item itself stays, dueAt unchanged).
- op=update RESETS fireCount/lastFiredAt on the affected trigger. Harmless for unlimited routines; slight progress loss for count-bounded ones.

Prefer op=add / op=update over op=replace_all for surgical edits — replace_all is heavier and harder for the user to reason about.

# 6. COMPLETE vs CANCEL — semantically opposite

- complete = "I did it / done" → marks the whole item done; all triggers disabled.
- cancel = "I don't need this / stop reminding me / delete it" → terminates the whole item; all triggers disabled; a routine stops firing entirely.

For a recurring routine the difference is mostly intent/wording, but both end the item. When the user just reports "did today's" on an ongoing routine, do NEITHER — acknowledge and let it keep firing.

# 7. CREATE — common pitfalls

- Trust dedup. An "already exists" result counts as success. Do NOT pre-call agenda_list to check.
- `triggers` is an array; multiple elements = multiple active schedules on one item ("9:00 AND 18:00", "1 day before AND at deadline"). Each spec carries its own action/message — different triggers on the same item can mix invoke / notify / notify_and_record.
- "1 hour from now", "100 days later" — compute the ISO yourself from the current time; trigger schemas only accept absolute ISO.
- Default `action` is notify (pure text ping, fire NOT recorded in history). Use notify_and_record when the user wants the fire to appear as an AI message in history (e.g. so a later reply like "已交" has context). Use invoke when the user wants the AI to actively work at fire time ("summarise the logs every morning at 8").

# 8. LIST — answering shape

Output is XML <agenda-list count="N"> with nested <trigger> elements. Times are UTC ISO truncated to minute. `expr` semantics: absolute=ISO; interval=ms string ("86400000"=24h); cron=6-field. Only enabled triggers are listed.
